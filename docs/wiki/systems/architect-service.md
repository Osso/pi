# Resident Architect Service

The resident Architect is a separate Pi process launched as a systemd user service. It observes the shared Pi control database, sends only material observations to `openai-codex/gpt-5.6-sol`, and delivers advisory hypotheses through targeted runtime mailbox messages. It does not dispatch work, issue operational directives, or make changes itself.

## Observation loop

`runArchitectService()` creates an `ArchitectObserver` and evaluates it every 30 seconds. The observer opens the control SQLite database read-only, configures the connection-local busy timeout, uses SQL to prefilter at most 20 current main sessions to a current main listener with matching fresh `ok` health, and reads up to 20 pending rows from `architect_requests`. It excludes subagents and the Architect itself, retains the most recently registered main-session listener for each live process, and uses deterministic metadata ordering for duplicate session IDs. Historical sessions that share a PID do not reappear as current work. Live runtimes refresh their listener and health every 60 seconds, keeping long-running sessions inside the five-minute freshness window.

The model receives only the bounded structured session projection, not raw listener or health fields. That snapshot is the Architect's sole session-inventory source by policy. `list_sessions` is also current-binding-aware, but remains blocked so Architect conclusions use the bounded snapshot that triggered the observation.

The first read produces an initial session-state observation. Later reads produce an observation when the ordered session snapshot changes, including goal JSON, or when `architect_requests` contains pending work. The `ask_architect` tool is the only supported request API and is limited to main sessions. Each new request stores either the absolute `project_path` supplied to `ask_architect` or, when omitted, the sender's current session cwd. Agents working in isolated worktrees pass that worktree path so Architect reads the branch's specs rather than the canonical checkout. Existing rows created before that column was added remain valid with no cwd. The observer atomically leases pending requests so competing Architect processes cannot duplicate them; expired leases return to pending. A request is completed only after `send_agent_message` successfully enqueues direct runtime-mailbox transport to its originating session. The Architect runtime disables all inbound runtime mailbox and shared-channel delivery, preventing other prompt sources from bypassing or racing the dedicated observer. Routine unchanged observations do not call the model. The prompt contains the structured snapshot and forbids querying `list_sessions`, so all inventory conclusions stay anchored to the bounded observation that triggered the turn.

## Advisor session and policy

The service reopens the newest `*_architect.jsonl` transcript under `<agent-dir>/architect-sessions/`, or creates it on first startup. This keeps the Architect's transcript separate from normal user sessions while its observer reads the same global control database. The transcript is nevertheless persisted into that control database as an archived non-subagent session: it is not in the Current Folder or All directory scopes, but it remains on disk and available in the Archived scope while the service continues using the same transcript.

The service requires the `openai-codex/gpt-5.6-sol` model. It loads shared user rules plus `rules/architect/*.md`; it does not load `rules/main/*.md`. Its system prompt permits normal tools when useful, but prohibits agent dispatch, file edits, session restarts, and autonomous remediation. Findings are advisory hypotheses for the receiving session to verify, accept, reject, or defer: the Architect must not issue commands, execution gates, stop conditions, operational sequencing, mandatory next steps, or reinterpret goal completion. For explicit project design or behavior requests, the Architect calls its host-side `read_project_spec` tool. The tool walks upward from the request cwd to find `docs/specs/README.md`, resolves symlinks, then permits only canonical Markdown paths inside that `docs/specs/` tree. Prompt policy tells the model to read the index first and only the relevant feature spec; the tool enforces path scope, not reading order or semantic relevance. This supports projects outside the Architect Bubblewrap workspace without mounting the host filesystem or copying spec text. A legacy request without cwd is answered with an explicit project-context limitation rather than inferred context. `goal_json.completedAt` means only that autonomous goal continuation has completed; it is not session-termination evidence. Consecutive control-SQLite snapshots can retain the same active main-session IDs while a goal is paused or completed. Architect uses session membership in the prefiltered bounded snapshot, never goal fields, as liveness evidence; it cannot inspect listener or health evidence directly. Sessions omitted because `is_subagent = 1` are intentionally excluded from Architect's main-session inventory. Explicit requests always receive a targeted response. Periodic advice remains limited to high-confidence conflicts, drift, or blockers. Responses use `send_agent_message` with the originating session ID; tool policy blocks `broadcast` and global `channel_post` fanout.

## Tool isolation

The service uses in-memory settings with `sandboxProfile: "read-only"` and loads the bwrap, agents-mailbox, and default Pyrun extensions. Pi retains its standard tools except Architect-disabled `ask_architect`, `broadcast`, and `contact_parent`, plus the Architect-only canonical-path-constrained `read_project_spec` host tool, while the bwrap extension routes file-tool workers, `bash`/`user_bash`, and Pyrun's runner process through Bubblewrap with the workspace mounted read-only. Pyrun is available in this mode, but its Pi bridge is disabled: it receives no Pi capability snapshot and cannot make bridge requests. The removed `contact_supervisor` name is not a compatibility alias.

Enabled host-side extension tools and hooks remain trusted capabilities outside that selected worker-routing boundary. This is not a confidentiality boundary against the trusted Architect or trusted enabled extensions. It limits selected worker filesystem/process access and prevents sandboxed runtimes from using Pi bridge capabilities. See [bwrap-sandbox.md](bwrap-sandbox.md) for mount and environment details.

## Deployment

`deploy.sh` builds and installs the platform-specific compiled `pi` binary, links it at the configured deployment bin directory, and renders `packages/coding-agent/systemd/pi-architect.service` into the user systemd directory with that exact binary path. The unit has restart-on-failure behavior and is enabled and started by deployment. Deployment reloads systemd, enables and starts the unit, explicitly restarts it so a newly installed binary is active, then requires `systemctl --user is-active --quiet pi-architect.service` to succeed.
